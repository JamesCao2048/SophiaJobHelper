# 2026 第二职业计划书：Agentic Efficiency Research → Product → Service
文档来源： https://github.com/JamesCao2048/personal-branding/blob/master/docs/second-career-plan-2026.md

> **核心定位**：以 Research 为起点，用"写书+开源"建立认知权威，逐步拓展到产品和跨领域咨询服务。

---

## 一、背景与优势分析

### 你的牌

| 维度 | 优势 |
|------|------|
| **第一职业** | 公司内用 Agent 做算子开发——你是 practitioner，懂成本、延迟、底层优化 |
| **学术背景** | 夫妻组合：AI + 软工 + HCI，覆盖 agentic workflow 的完整链条 |
| **时代背景** | 2026 SaaSpocalypse 刚发生，市场混乱期 = 认知套利窗口期 |
| **已有资产** | SeedGather（内容采集）、snotes（知识管理）、OpenClaw 实战经验 |
| **Sophia** | 找教职中，agentic workflow 的 HCI 研究方向正热 |

### 为什么 Research-First

1. **市场还在早期**——真正的 agentic-first 产品形态还没有共识，现在冲进去做产品大概率做错方向
2. **代码会贬值，认知在升值**——报告说的"Sloppable"：如果护城河只是代码，AI 会轻易复制
3. **你的比较优势是"理解力"而非"开发速度"**——算子开发 + HCI 的交叉视角，比纯 vibe coding 值钱得多
4. **Research 可以拓展到非 CS 领域**——效率工程的方法论是通用的，未来可以切入学术、法律、金融等垂直领域

## 二、三阶段路线图

### Phase 1：边写书边建影响力（现在 → 3-6 个月）

**核心目标**：建立"Agentic Efficiency 领域专家"的认知品牌

#### 2.1 写书：《让 AI 替你忙》= Research + Brand Building + 市场调研

- 每一章都是一个领域的深度调研
- 写的过程中自然发现"这个痛点没人解决"
- **Book-Driven Development (BDD)**：每章配套开源代码/MCP Server/Workflow 脚本
  - 例：信息过滤章 → 配套 Gmail/Slack 降噪 Agent 开源代码
  - 例：知识管理章 → 配套 Zotero-to-Notion 连接器
- GitHub star 数 = 需求验证信号（哪个模版最受欢迎 → 哪个值得产品化）

#### 2.2 内容分发：公众号 + 小红书连载

- 连载频率：每周 1-2 篇
- 内容策略：方法论 + 实操案例 + 开源工具介绍
- **受众反馈 = 最便宜的市场调研**

#### 2.3 即时变现（不等产品做完就能赚的钱）

| 变现方式 | 说明 | 预期收入 |
|----------|------|----------|
| **书本身** | 出版/连载/付费专栏 | 长尾收入 |
| **咨询服务** | 用 Agent 做算子开发的实战经验，帮团队搭建 agentic workflow | $2000-5000/单 |
| **课程/Workshop** | "如何用 OpenClaw 搭建个人效率系统" | 录一期反复卖 |
| **付费代码库** | 书配套的开箱即用 Agent 模版（GitHub Sponsors / 付费专栏） | $50-100/份 |

#### 2.4 学术产出（Sophia 方向）

- Sophia 找教职可以投 agentic workflow 方向的 paper
- 目标会议：CHI / CSCW / UIST
- **你们自用工具 + 使用数据 = 现成的研究素材**
- Paper 角度："We built and used X tool for Y months, here's what we learned about agentic workflow in practice..."
- 一鱼三吃：自用、开源、发 paper

---

### Phase 2：自用工具 → 开源生态（3-6 个月）

**核心目标**：从自己的痛点中长出工具，开源建立社区

#### 2.5 Dogfooding：Sophia 找教职 = MVP 测试场

把 Sophia 找教职过程中的每个痛点都用 Agent 解决：
- 自动爬取学校职位信息
- 根据学校偏好自动调整 CV 关键词
- 管理推荐信发送状态
- 针对不同学校定制 Cover Letter
- **这些工具自用验证后开源**

#### 2.6 自用工具矩阵（已有 + 待建）

| 工具 | 状态 | 方向 |
|------|------|------|
| **SeedGather** | ✅ 已有 | 多平台内容采集 → 信息监控 Agent |
| **snotes** | ✅ 已有 | 碎片想法收集 → 主动式知识连接器 |
| **Academic Career Agent** | 🔨 待建 | 学术求职全流程自动化 |
| **内容分发 Agent** | 🔨 待建 | 写稿→适配格式→排版→发布→追踪 |
| **Deep Context Agent** | 🔨 待建 | 连接所有数字足迹，主动整理+提示 |

#### 2.7 开源策略

> **开源的不是"产品"，开源的是"能力"。付费的是"省心"。**

- 代码框架开源 → 社区贡献 → star 数验证需求
- 配置好的、持续更新的版本 → 付费订阅
- 有人反复问"能不能帮我部署？" → 托管版/SaaS 的起点

#### 2.8 差异化：高效能 Agent（算子开发优势）

- 主打 **On-Device Agentic Efficiency**（本地运行，不传数据）
- 卖点："别人的 Agent 跑一次 $0.5 还要上传数据；我的 Agent 在你本地跑，不传数据，速度快 10 倍"
- 对学术界（数据敏感）和极客圈（在乎性能）有很强吸引力

---

### Phase 3：产品化 + 跨领域拓展（6 个月+）

**核心目标**：从验证过的需求中提炼付费产品，拓展到非 CS 领域

#### 2.9 产品化信号

当出现以下信号时，开始产品化：
- GitHub 上某个模版反复被 fork/star
- 咨询中同一个需求被反复提出
- 有人主动问"有没有托管版？"
- 读者在评论区反复问同一个问题

#### 2.10 产品形态：Service-as-a-Software

不卖工具，卖结果：

| 传统 SaaS | Agentic Service |
|-----------|-----------------|
| "AI 写作助手" $9.9/月 | "交付完成的 Grant Proposal 格式化" 按份收费 |
| "内容管理工具" 按席位 | "每月交付的洞察报告" 按结果收费 |
| "闲鱼分析软件" 订阅制 | "帮你多赚 X 元" 按成交抽成 |

#### 2.11 跨领域 Research 拓展

效率工程方法论可以推广到：
- **学术科研**：论文投稿 Agent、文献综述 Agent、Grant 申请 Agent
- **法律**：合规检查 Agent、合同审查 Agent（监管护城河）
- **金融**：财务报告自动化（报告中的"Excel 时刻"）
- **医疗**：临床数据处理、病历结构化
- **关键**：每个领域的"隐性知识"和"合规要求"都是通用 AI 搞不定的护城河

## 三、关键原则

### 3.1 不要做的事

- ❌ 现在花 3 个月全职写一个通用 SaaS
- ❌ 按席位收费的订阅模式
- ❌ 做"通用的个人 AI 助手"（Claude 的领地）
- ❌ 只有代码没有方法论的产品（会被 Vibe Coding 吃掉）

### 3.2 必须做的事

- ✅ 先卖认知，再卖代码
- ✅ 从自己的真实痛点出发（Dogfooding）
- ✅ 按结果/用量定价，不按人头
- ✅ 脏活就是护城河（平台适配、合规、数据管道）
- ✅ Agent-native 架构（从第一天就设计成 API/MCP 接口）
- ✅ 保持学术产出（一鱼三吃：自用、开源、发 paper）

### 3.3 心态管理

> **"研究"不等于"浪费时间"。在 Agent 时代，深刻理解业务逻辑 + 底层技术实现能力，比单纯的"产品经理思维"重要得多。**

- 你现在的"研究"，只要是基于真实痛点的代码实践，就是在做产品
- 书，是你产品的"说明书"和"营销文案"
- 写书 + 嵌入代码 = 风险最低、收益最高的路径
- 不要焦虑"错过机会"——90% 的 agentic 产品会在 6 个月内被通用 AI 更新吃掉，能活下来的需要认知深度
